This is Alfred V1, he isn't very smart nor particularly good at what he is supposed to do.

I made this robot for the RIT Annual Engineering competition in November of 2017.

He is run on a Raspberry Pi 3, Python code, 3 Ultrasonic Sensors; 1 Front and 2 facing ~45 Degrees from the front side, a small battery bank, 5 AA Rechargeable Batteries and a Dual Full Bridge L298N.

There were alot of issues with this version, one that always occurred  is that since it is slow Python code it may not catch the signal from the sensor thus it keeps waiting and waiting and waiting, the robot hangs and repeats whatever it was last command was usually going forward. So when this error occurred it would just ram into the wall.

Another error was the actual built of the robot. Since I didn't give myself a lot of time to build this robot (made it in a few days before the competition) I didn't really make some wooden structure that holds the sensors. Instead I do what any Engineer loves to do, use something super modular and easy to chance. I used Legos. So as a added bonus when the robot does ram itself into a wall at least it doesn't damage the sensors too much, because as you saw in the video they just fall off. Now if I Super Glued the Legos, then it may not have been total garbage.

This is Alfred V2, a bit smarter and much better at what he is supposed to do.

Contrary to the video, he is technologically superior in every way compared to Alfred V1. Mainly he is now on Arduino. That means low level code, directly on 'bare metal'. 

Though as great as he is, he isn't perfect yet. Of course when developing anything each revision does have improvements but also faults.

One of the faults that I recognize now is that I have been taking this approach from a Software Engineer, rather a Mechatronics Engineer. What I mean by that is when doing software, you know it can execute your code as is. So there isn't any random errors or chances involved. When handling this as a Mechatronics Engineer, you need to program your software a bit differently to take into consideration other factors. Hopefully, these changes will improve Alfred further towards V3!
